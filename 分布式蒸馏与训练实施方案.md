# SwarmForge 技术实施方案

## 1. 系统概述

SwarmForge 分两个阶段建设：

**Stage 1 — 训练网络**：全球节点加入网络，贡献算力参与模型训练（蒸馏、RL、微调），获得 $FORGE Token 激励。类似以太坊早期矿工挖矿。

**Stage 2 — 推理网络**：当模型能力达标后，开放全球推理服务。用户消耗 $FORGE Gas 发起推理请求，多个节点协作完成，节点获得 Gas 收入。

**核心逻辑**：先集体训练出足够强的模型，再开放服务。不是先做产品找用户，是先建网络让网络产出产品。

```
以太坊范式:
  矿工网络 → 出块 → 链有价值 → 开放智能合约 → 生态

SwarmForge 范式:
  训练网络 → 出模型 → 模型有价值 → 开放推理服务 → 生态
```

## 2. 为什么是 7B 专家集群

### 2.1 数据支撑

| 方案 | HumanEval | 单次延迟 | 内存占用 |
|------|-----------|---------|---------|
| 7B 单模型（单次） | 73.2% | ~0.5s | 5GB |
| 32B 单模型（单次） | 84.8% | ~3s | 20GB |
| Opus 4.6（单次） | ~93% | ~5s | 云端 |
| **7B 管线（20 候选，单机）** | **94.5%** | **16s** | **5GB** |
| **7B 管线（20 候选，网络）** | **94.5%+** | **~2s** | **5GB/节点** |
| 32B 管线（5 候选，单机） | 95.1% | 43s | 20GB |

### 2.2 选 7B 的本质原因：最大化矿工数量

| 维度 | 7B 专家集群 | 单个 70B |
|------|------------|---------|
| 入网门槛 | 8GB（任何笔记本） | 48GB（极少数设备） |
| 潜在矿工数 | **数百万** | 几千 |
| 推理延迟 | 0.5s / 节点 | 3-5s / 节点 |
| 训练成本 | LoRA $0-50 / 专家 | 全量 $5000+ |
| 网络效应 | 门槛低 → 矿工多 → 训练数据多 → 模型强 | 门槛高 → 人少 → 增长慢 |

和以太坊一样：网络越大越强。7B 是唯一能让网络达到万人规模的选择。

## 3. Stage 1：训练网络

### 3.1 节点角色 = 矿工

节点加入网络的核心动作是参与训练，不是提供推理。推理是后面的事。

```
矿工的工作:

1. RL 数据采集（所有 Tier 都能做）
   - 从网络领取编程题
   - 本地 7B 做题 → sandbox 自动评测
   - 上报: (题目, 代码, pass/fail, 轨迹)
   - 获得 $FORGE 奖励

2. SFT 蒸馏采集（Tier 1/2 节点 或 用户自带 API Key）
   - 从网络领取蒸馏任务
   - 调用本地大模型(32B/72B) 或 前端 API 生成高质量回答
   - 上报: (问题, 思考过程, 解答)
   - 获得 $FORGE 奖励（SFT 数据价值更高 → 奖励更多）

3. LoRA 训练（Tier 1 节点）
   - 从网络下载训练数据集
   - 本地跑 LoRA SFT / RL 训练
   - 上传训练好的 LoRA 权重
   - 获得大量 $FORGE 奖励
```

### 3.2 训练任务调度

```
训练调度器（协议层）:

1. 维护全局训练计划:
   - 当前 Epoch 目标（7 / 21 / 50 个专家）
   - 每个专家还需多少 SFT / RL 数据
   - 当前进度百分比

2. 任务分发:
   POST /mining/task → {type: "rl", problem: "...", tests: "...", reward: 5}
   POST /mining/task → {type: "sft", problem: "...", teacher_model: "32b", reward: 15}
   POST /mining/task → {type: "train", dataset_url: "...", expert_id: "gen-v0", reward: 100}

3. 数据验证:
   - 每条 RL 数据 → sandbox 验证 pass/fail
   - 每条 SFT 数据 → 冗余验证 + Tier 1 抽检
   - 验证通过 → 入库 + 发 Token
   - 验证失败 → 拒绝 + 扣信誉

4. 训练触发:
   - 某专家数据量达标 → 触发训练
   - Tier 1 节点竞争训练权（质押 $FORGE）
   - 训练完成 → 评测 → 达标则全网推送新 LoRA
```

### 3.3 节点分层

| 层级 | 设备 | 内存 | 矿工角色 | 挖矿效率 |
|------|------|------|---------|---------|
| **Tier 1** | M4 Max, A100 | 64GB+ | RL + SFT 蒸馏 + LoRA 训练 | 最高 |
| **Tier 2** | 4090, M2 Pro | 24-48GB | RL + SFT 蒸馏 | 高 |
| **Tier 3** | 3060, M1, 16GB | 16GB | RL 数据采集 | 中 |
| **Tier 4** | 8GB 笔记本 | 8GB | 简单 RL 采集 | 低 |

### 3.4 数据采集规模

```
100 个矿工节点 × 4h/天:
  RL: ~120K 条/天
  SFT: ~15K 条/天（受 Tier 1/2 数量限制）

1,000 个矿工节点 × 4h/天:
  RL: ~1.2M 条/天
  SFT: ~100K 条/天

10,000 个矿工节点 × 4h/天:
  RL: ~12M 条/天
  SFT: ~500K 条/天

对比:
  Epoch 1 全部训练数据 = 241K 条
  → 100 个矿工 2 天就能采完
  → 1,000 个矿工几小时就能采完

网络的力量：单人需要 5 天采集的数据，1,000 人几小时搞定。
```

### 3.5 数据质量保障

| 机制 | 说明 | 类比 ETH |
|------|------|---------|
| **Sandbox 验证** | 代码必须通过执行验证 | 类似区块验证 |
| **冗余验证** | 同一题发 3 个矿工，2/3 一致采纳 | 类似共识机制 |
| **Tier 1 抽检** | 高级节点用 32B 抽检 10% 数据 | 类似验证者 |
| **信誉系统** | 高信誉数据优先，低信誉降权 | 类似质押权重 |
| **对抗检测** | 检测批量相似提交 | 类似反女巫 |

### 3.6 模型训练执行

**Epoch 1-2（半中心化训练）**：
```
数据采集: 去中心化（所有矿工）
数据汇总: 去中心化存储（IPFS）
训练执行: Tier 1 节点竞争（质押 $FORGE 获得训练权）
评测验证: 多个 Tier 1 节点独立评测
LoRA 推送: P2P 分发给所有节点
```

**Epoch 3+（去中心化训练）**：
```
联邦学习: 每个节点本地微调 → 上传梯度 → 协议层聚合
完全去中心化: 节点间直接交换梯度（研究中）
```

## 4. $FORGE Token 经济模型

### 4.1 参考以太坊设计

| ETH 概念 | SwarmForge 对应 |
|----------|----------------|
| PoW 挖矿 → PoS | 训练挖矿（早期）→ 质押验证（后期）|
| Gas Fee | 推理 Gas（用户消耗 Token 发起请求）|
| 区块奖励 | 训练奖励（每 Epoch 释放 Token）|
| 质押 | 节点质押 $FORGE 获得训练/推理资格 |
| Slash | 提交垃圾数据 / 作弊 → 扣除质押 |
| EIP 治理 | 网络投票决定训练方向、模型升级 |

### 4.2 Token 发行

```
总量: 1,000,000,000 $FORGE (10 亿)

分配:
  训练挖矿:    600,000,000 (60%) — 按 Epoch 线性释放
  推理奖励:    200,000,000 (20%) — 推理网络开放后释放
  核心开发:    100,000,000 (10%) — 4 年线性解锁
  生态基金:    100,000,000 (10%) — DAO 治理

通胀: 无。总量固定。
减半: 每 6 个月训练挖矿奖励减半（类似 BTC 减半）
```

### 4.3 挖矿奖励

| 训练贡献类型 | 单次奖励 ($FORGE) | 说明 |
|------------|-----------------|------|
| RL 数据采集（通过验证） | 5 | 任何节点都能做 |
| SFT 蒸馏数据（通过验证） | 15 | 需要大模型 |
| LoRA 训练完成（通过评测） | 500 | Tier 1 节点 |
| 发现数据质量问题 | 10 | 审计节点 |

### 4.4 推理 Gas 费

```
推理网络开放后:

用户请求:
  简单任务（补全）: 1 $FORGE Gas
  中等任务（函数生成）: 5 $FORGE Gas
  复杂任务（多文件）: 20 $FORGE Gas

Gas 分配:
  70% → 执行推理的节点
  20% → 网络维护（协议层）
  10% → 训练基金（持续训练新专家）
```

### 4.5 质押机制

```
成为矿工:
  质押 100 $FORGE → 获得 Tier 4 矿工资格
  质押 1,000 $FORGE → 获得 Tier 3 矿工资格
  质押 10,000 $FORGE → 获得 Tier 1 训练资格

惩罚 (Slash):
  提交垃圾数据 → 扣 10% 质押
  连续 3 次数据被拒 → 扣 30% 质押 + 暂停 24h
  恶意攻击 → 全额没收

质押好处:
  - 获得训练任务优先权
  - 获得更高挖矿奖励乘数
  - 参与网络治理投票
```

### 4.6 飞轮

```
矿工加入 → 质押 $FORGE → 贡献训练数据 → 获得 $FORGE
    ↓
模型变强 → 推理服务有价值
    ↓
用户付 Gas 使用推理 → $FORGE 有真实需求
    ↓
$FORGE 增值 → 更多矿工加入
    ↓
更多训练数据 → 模型更强 → ∞
```

## 5. Stage 2：推理网络

### 5.1 开放条件

推理网络不是一开始就有。满足以下条件后开放：

```
✅ HumanEval 管线 ≥97%（训练后的专家）
✅ 至少 21 个专家覆盖核心能力×领域
✅ 至少 1,000 个在线节点
✅ 网络延迟 P90 < 3 秒
✅ 网络投票通过（$FORGE 持有者投票）
```

### 5.2 多节点协作推理管线

```
用户请求: "写一个 REST API 的用户认证模块"
  │ 消耗 5 $FORGE Gas
  ▼
[协议层] 路由 → 选择全球 5-6 个节点:

Stage 1: 并行生成 (~1s)
  🇯🇵 节点A [gen-web]: 候选 1,2,3     ← 获得 0.5 $FORGE
  🇩🇪 节点B [gen-web]: 候选 4,5,6     ← 获得 0.5 $FORGE
  🇧🇷 节点C [gen-web]: 候选 7,8,9     ← 获得 0.5 $FORGE

Stage 2: 测试生成 (并行)
  🇰🇷 节点D [test-web]: 测试用例       ← 获得 0.5 $FORGE

Stage 3-4: 验证 + 修复
  🇺🇸 节点E [fix-web]: 修复 + 验证     ← 获得 0.7 $FORGE

Stage 5: 审查选择
  🇮🇳 节点F [review-web]: 选最优       ← 获得 0.5 $FORGE

用户拿到结果。~2 秒。消耗 5 $FORGE。
6 个节点共获得 3.2 $FORGE（剩余归网络维护+训练基金）。
```

### 5.3 延迟分析

| 阶段 | 单机串行 | 网络并行 | 加速比 |
|------|---------|---------|--------|
| 生成 9 候选 | 4.5s | 0.8s | 5.6× |
| 生成测试 | 0.5s | 与生成并行 | ∞ |
| 验证 | 2s | 0.5s | 4× |
| 修复 | 1s | 0.5s | 2× |
| 选择 | 0.5s | 0.3s | 1.7× |
| **总计** | **~9s** | **~2s** | **4.5×** |

### 5.4 容错

```
节点超时: 自动转发备选节点，用户无感知
垃圾结果: sandbox 验证过滤，节点信誉 -1
全部失败: 触发 Tier 1 节点（32B 兜底）
```

## 7. 通信协议

### 7.1 节点注册与发现

```
注册:
  POST /register {
    node_id,
    tier,
    stake_amount,      // 质押数量
    hardware_info,     // 自动检测
    available_experts, // 当前已有的专家
    bandwidth,
    location           // 地理位置（用于就近路由）
  }

发现:
  DHT (Kademlia) P2P 发现
  Bootstrap 节点用于初始连接
```

### 7.2 训练协议

```
领取任务:
  GET /mining/task → {type, problem, tests, reward, deadline}

提交结果:
  POST /mining/submit {
    task_id,
    node_id,
    result: {code, pass_result, trajectory},
    signature  // 签名防伪
  }

奖励结算:
  → 验证通过 → $FORGE 转入矿工账户
  → 验证失败 → 拒绝 + 扣信誉
```

### 7.3 推理协议（Stage 2 开放后）

```
用户请求:
  POST /infer {
    prompt,
    complexity,        // 简单/中等/复杂
    max_gas,           // 愿意支付的最高 Gas
    timeout_ms
  }

节点响应:
  {result, confidence, exec_time, node_id}

Gas 结算:
  请求完成 → Gas 自动分配给参与节点
```

## 8. 隐私与安全

### 8.1 训练阶段隐私

```
训练数据不涉及用户私有代码:
  - SFT 数据: 公开编程题 + 大模型回答
  - RL 数据: 公开编程题 + 本地模型做题
  - 没有敏感信息
```

### 8.2 推理阶段隐私

```
方案 A: 代码匿名化
  用户本地替换变量名/类名 → 发到网络 → 拿到结果后还原

方案 B: 信任分层
  通用请求 → 网络推理
  私有代码 → 仅本地推理（用已下载的专家）

方案 C: 加密推理（长期研究）
  同态加密 / 安全多方计算
```

### 8.3 防攻击

| 攻击 | 防御 | 类比 ETH |
|------|------|---------|
| 女巫攻击 | 质押 + 设备指纹 | PoS 质押门槛 |
| 数据投毒 | 冗余验证 + sandbox + 抽检 | 共识验证 |
| 51% 攻击 | Tier 1 节点多方验证 | 验证者分布 |
| DDoS | 请求需要 Gas / 质押门槛 | Gas 机制 |
